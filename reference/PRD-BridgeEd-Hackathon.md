# PRD：BridgeEd（Hackathon 版）

## 1. 文档信息
- **产品名称**：BridgeEd
- **目标形态**：Web（三端：老师端 / 家长端 / 学生端）
- **约束条件**：必须使用 CurricuLLM API 进行 GenAI 能力实现

## 2. 背景与问题定义

当前家校沟通存在明显断层：  
老师提供的信息常偏“教学语言”，家长不易理解、更难执行；家长有反馈需求但老师时间有限；不同语言和文化背景进一步放大沟通成本，导致“课堂所学”无法高质量延伸到家庭学习场景。

**核心问题**：如何把“老师可发布的信息”转化为“家长可执行的行动”，并形成低负担的双向反馈闭环。

## 3. 产品目标（Hackathon MVP）

### 3.1 业务目标
- 提升家长对课堂内容的理解与可执行度
- 降低老师沟通成本，而非增加额外负担 (这一点要在pre的时候强调， 我们在设计的时候不能给教师端特别heavy的操作)
- 证明三端协同闭环可行（老师发布 -> 家长行动 -> 学生反馈 -> 老师调整） （pre的时候强调有闭环）

### 3.2 用户目标
- **老师**：1分钟内完成一次有效发布，能快速看到家长/学生反馈
- **家长**：看得懂、做得到、遇到问题可即时获得 AI 帮助
- **学生**：完成轻量练习并反馈学习感受

### 3.3 Hackathon 目标
- 完成可演示原型
- 具备 2-3 分钟可讲清的“闭环故事”
- 明确体现 CurricuLLM API 的使用价值

## 4. 目标用户与场景

### 4.1 目标用户
1. **老师**（主导发布与查看反馈）
2. **家长/监护人**（核心执行者）
3. **学生**（完成练习与状态反馈）

### 4.2 核心场景
1. 老师发布本周/当天学习目标，家长收到“可执行版”内容
2. 家长不理解概念时，AI 自动解释并提供家庭辅导建议
3. 学生完成配套小练习并提交心情/理解反馈
4. 老师查看班级汇总并进行二次说明或个别跟进

## 5. 价值主张

- 对老师：**高效触达 + 结构化反馈 + 可快速干预**
- 对家长：**降低理解门槛 + 明确行动步骤 + 及时求助**
- 对学生：**轻量练习 + 情绪表达 + 学习参与感**
- 对学校场景：**增强家校协同质量，减少无效信息传递**

## 6. MVP范围（Must-have / Nice-to-have）

### 6.1 Must-have (P0)
1. **老师一键发布学习卡**
   - 输入：课堂知识点/目标（可文本输入）
   - 输出：家长友好总结 + 3条家庭可执行建议 + 可选多语言版本
2. **家长 AI 解释助手**
   - 家长可对学习卡点击“没看懂”
   - AI 提供简化解释、例子、术语翻译
3. **家长一键生成学生小练习**
   - 针对当前知识点自动生成 2-3 题（含参考答案/讲解）
4. **结构化双向反馈**
   - 家长反馈模板：是否完成、困难点、需要老师帮助
   - 老师端查看聚合反馈而非碎片聊天
5. **学生端轻量提交**
   - 完成练习 + 心情/理解度打卡
   - 结果回流到老师/家长视图

### 6.2 Nice-to-have（P1）
- 老师与家长实时聊天/语音视频通话预约
- 自动视频讲解生成
- 长周期成长报告
- 更深度个性化学习路径

## 7. 功能需求（按端划分）

### 7.1 老师端（Teacher Portal）
#### 功能 T1：发布学习卡
- 新建发布（班级/单个家长）
- 输入课堂内容（知识点、作业目标）
- 调用 CurricuLLM API 生成：
  - 家长易懂摘要
  - 家庭辅导动作建议（3条）
  - 多语言版本（如中/英）
- 发布并推送给目标家长

#### 功能 T2：反馈看板
- 查看每条学习卡的家长完成率
- 查看高频困难点（聚合）
- 查看学生练习完成状态与情绪分布

#### 功能 T3：快速跟进
- 基于困难点生成“二次说明”
- 向班级或个体补充提醒

### 7.2 家长端（Parent Hub）
#### 功能 P1：学习卡查看
- 查看老师推送的学习内容
- 显示“今天回家可以做什么”

#### 功能 P2：AI 解释与翻译
- 点击“我没看懂”获取多层解释（简版/举例）
- 一键切换语言

#### 功能 P3：家庭行动与出题
- 根据知识点生成家庭互动建议
- 一键生成给孩子的小练习（2-3题）

#### 功能 P4：结构化反馈
- 反馈是否已执行、遇到什么困难、是否需要老师联系

### 7.3 学生端（Student Mini）
#### 功能 S1：练习作答
- 完成家长发起的小练习
- 提交答案

#### 功能 S2：学习感受打卡
- 情绪选择（如 🙂/😐/🙁）
- 一句话反馈“哪里不懂”

## 8. CurricuLLM API 使用方案（必须项）

至少覆盖以下 4 类调用（Demo 中可见）：
1. **课程语言转家长语言**（简化/重写）
2. **多语言翻译**（面向 EAL/D 家庭）
3. **家庭行动建议生成**（可执行步骤）
4. **学生练习题生成**（题目+参考答案/讲解）

> 备注：所有 AI 输出需保留“教师可复核”入口，避免直接不可控发送。

## 9. 关键用户流程（E2E）

1. 老师创建并发布学习卡
2. 家长收到后阅读，若不懂则调用 AI 解释
3. 家长生成小练习并给学生完成
4. 学生提交答案与心情
5. 家长提交结构化反馈
6. 老师在看板看到结果并进行下一步教学沟通

## 10. 非功能需求

- **可用性**：核心流程 3 次点击内可到达主要动作
- **性能**：AI 生成响应目标 < 8 秒（Hackathon 可接受）
- **安全与隐私**：最小化存储学生敏感信息；演示数据可脱敏
- **可维护性**：模块化三端结构，便于后续扩展

## 11. 成功指标（Demo + 早期验证）

### 11.1 演示级指标（Hackathon）
- 完整跑通至少 1 条学习卡闭环流程
- 3 端都能展示可操作界面
- CurricuLLM API 调用在演示中可见且稳定

### 11.2 产品级指标（若继续）
- 家长学习卡阅读后行动率（点击“已执行”）
- 家长 AI 解释使用率
- 学生练习完成率
- 老师二次跟进耗时下降（相对传统群聊）

## 12. 里程碑（建议 3 天冲刺）

- **Day 1**：确定信息架构 + 三端低保真 + API 接口打通
- **Day 2**：实现核心功能（T1/P2/P3/S1）+ 数据回流
- **Day 3**：完善看板与反馈闭环 + 演示脚本 + Pitch Deck + 视频录制

## 13. 风险与应对

- **风险1：范围过大** -> 严控 MVP，仅做“发布-解释-练习-反馈”闭环
- **风险2：AI 输出不稳定** -> 增加模板与教师复核机制
- **风险3：三端开发压力** -> 学生端做轻量页面，不做复杂社交功能
- **风险4：演示时间不足** -> 提前固化一条 demo 数据链路

## 14. 本次不做（Out of Scope）

- 实时音视频通话系统
- 完整 IM 聊天系统（先保留结构化反馈）
- 高复杂度长期学习画像
- 全学段/全学科深度适配
